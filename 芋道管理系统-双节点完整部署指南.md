# 芋道管理系统 · 双节点完整部署指南

> 操作系统：macOS（Mac mini）  
> 部署时间：2026 年 6 月  
> 目标：在一台 Mac mini 上，通过 Docker + 宿主机 Node 同时运行两套独立的 yudao 管理系统

---

## 一、系统架构总览

```
┌─────────────────────────────────────────────────────────┐
│                     Mac mini (宿主机)                   │
│                                                         │
│  :48080 ──── yudao-server-a (后端A / yudao-boot-mini)  │
│  :48081 ──── yudao-server-b (后端B / ruoyi-vue-pro)     │
│  :8080  ──── ruoyi-admin-b (前端B / vue3 + nginx)       │
│  :8081  ──── vben-admin-a (前端A / vben5 + nginx)       │
│  :3306  ──── yudao-mysql                                │
│  :6379  ──── yudao-redis                               │
└─────────────────────────────────────────────────────────┘
```

**系统 A**：yudao-boot-mini（后端）+ vben5（前端）  
**系统 B**：ruoyi-vue-pro（后端）+ yudao-ui-admin-vue3（前端）

两套系统共用宿主机已有的 MySQL 和 Redis，不创建新数据库容器。

## 二、前置条件

### 2.1 Docker 环境

Docker Desktop 必须运行，Docker 命令完整路径：

```bash
/Applications/Docker.app/Contents/Resources/bin/docker
```

> **注意**：终端里直接敲 `docker` 可能找不到命令，需要用完整路径，或者把 Docker 二进制目录加入 `PATH`：
> ```bash
> export PATH="/Applications/Docker.app/Contents/Resources/bin:$PATH"
> ```

```bash

```

```bash
# 之后所有 docker 命令加 --context hermes2
docker --context hermes2 ps
```

推荐 Node 22+，pnpm 9+。

### 5.4 构建后端 Docker 镜像

**yudao-boot-mini 镜像：**

```dockerfile
# yudao-boot-mini/Dockerfile
FROM eclipse-temurin:21-jre

WORKDIR /yudao

COPY yudao-admin/target/yudao-admin.jar ./app.jar
COPY docker-entrypoint.sh ./docker-entrypoint.sh
RUN chmod +x docker-entrypoint.sh

EXPOSE 48080
ENTRYPOINT ["./docker-entrypoint.sh"]
```

**ruoyi-vue-pro 镜像：**

```dockerfile
# ruoyi-vue-pro/Dockerfile
FROM eclipse-temurin:21-jre

WORKDIR /yudao

COPY yudao/ruoyi-admin/target/ruoyi-admin.jar ./app.jar

EXPOSE 48080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

构建并打标签：

```bash
# 系统A后端镜像
docker build -t yudao-server-a ~/Documents/BX/yudao/yudao-boot-mini

# 系统B后端镜像
docker build -t yudao-server-b ~/Documents/BX/yudao/ruoyi-vue-pro
```

### 5.5 启动后端容器

> **关键**：容器的 `EXPOSE 48080` 是容器内部端口；宿主机的 `-p 48081:48080` 是把容器 48080 映射到宿主机 48081。**其他容器访问该后端时，用容器间真实端口（48080），不是宿主机映射端口（48081）。**

```bash
# 系统A后端
docker run -d \
  --name yudao-server-a \
  --network yudao-system_default \
  -p 48080:48080 \
  yudao-server-a

# 系统B后端
docker run -d \
  --name yudao-server-b \
  --network yudao-system_default \
  -p 48081:48080 \
  yudao-server-b
```

## 七、常见问题与解决

### Q1：pnpm install 报引擎版本不匹配

**原因**：Node/pnpm 版本不符合 `package.json` 中 `engines` 的要求。  
**解决**：修改 `package.json`，将 `engines` 中的版本要求放宽，然后用 `pnpm install --ignore-scripts`。

### Q2：postinstall stub 脚本报 `unrun` 模块找不到

**原因**：vben5 的构建 stub 脚本依赖 `unrun`，但 pnpm install 时未安装。  
**解决**：
```bash
pnpm add -w unrun
pnpm rebuild
```

### Q3：构建时报 `DICT_TYPE is not exported from 'vue'`

**原因**：源码里错误地从 `vue` 导入 yudao 内部常量（如 `DICT_TYPE`、`generateWmsCode`、`KeFuMessageContentTypeEnum` 等），这些不是 vue 的标准导出，应该从 `@vben/constants` 导入。  
**解决**：手动修改 import 语句，从 `vue` 中移除这些错误导入，改用空值或从正确包导入。

### Q4：nginx 代理 API 返回 502

**排查步骤**：

1. 确认两个容器在同一 Docker 网络：
   ```bash
   docker inspect <nginx容器> --format '{{range .NetworkSettings.Networks}}{{.NetworkID}}{{end}}'
   docker inspect <后端容器> --format '{{range .NetworkSettings.Networks}}{{.NetworkID}}{{end}}'
   ```

2. 从 nginx 容器内部测试连通性：
   ```bash
   docker exec <nginx容器> wget -q -O- http://yudao-server-b:48080/admin-api/system/auth/login
   ```

3. **核心问题**：`proxy_pass` 填的是容器内部端口（`48080`），不是宿主机映射端口（`48081`）。

### Q5：Docker Hub / GHCR 拉取镜像超时

**解决**：使用可访问的镜像源。验证可用镜像源：
```bash
docker pull quay.io/nginx/nginx-unprivileged:alpine   # ✅ 可用
docker pull docker.m.daocloud.io/library/nginx:alpine  # ✅ 可用
docker pull registry.npmmirror.com/library/nginx:alpine # ❌ 超时
docker pull docker.io/library/nginx:alpine             # ❌ 超时
```

### Q6：后端容器 OOM 被 kill

**解决**：增加 Java 堆内存限制：
```bash
docker run -e JAVA_OPTS="-Xmx512m -Xms256m" ...
```

### Q7：Git clone 失败（仓库不存在）

**原因**： `ruoyi-vue-pro` 不在 `yudaocode` 组织里，而在 `YunaiV` 个人账户下。  
**解决**：
```bash
git clone https://github.com/YunaiV/ruoyi-vue-pro.git   # 而非 yudaocode/ruoyi-vue-pro
```

## 十、关键经验总结

1. **容器间通信用内部端口，宿主机映射端口仅供宿主机访问**：后端容器 `EXPOSE 48080` 是内部端口，`-p 48081:48080` 是宿主机映射。前端 nginx 在容器内通过 `yudao-server-b:48080` 访问，而不是 `localhost:48081`。

2. **所有仓库 clone 到同一目录**：避免 Docker volume 挂载时路径混乱。

3. **网络不通时多试几个镜像源**：`quay.io` 和 `docker.m.daocloud.io` 在国内通常可用。

4. **vben5 构建需要特殊处理**：`--ignore-scripts` + `pnpm add -w unrun` + 修复错误 import 三步缺一不可。

5. **`ruoyi-vue-pro` 归属 YunaiV 而非 yudaocode**：clone 时用 `YunaiV/ruoyi-vue-pro` 而非 `yudaocode/ruoyi-vue-pro`。

6. **Java 后端 OOM**：给容器加内存限制，或加 `JAVA_OPTS="-Xmx512m -Xms256m"`。

## 十一、外网访问配置（外网域名 + 端口映射）

### 11.1 背景与目标

通过外网域名（如 `hermes.bongxin.com.cn:18081`）访问部署在内网的 yudao-vben 系统。前端和 API 共用同一域名，API 路径通过 nginx 代理到后端。

### 11.2 前端 API 路径动态化

**核心原则**：前端不写死任何 IP 或域名，API 路径始终使用相对路径，自动适配当前访问的 host 和端口。

修改 `yudao-ui-admin-vben/apps/web-antd/src/api/request.ts`：

```typescript
// baseURL 改为空字符串，请求走相对路径（window.location.origin）
export const requestClient = createRequestClient('', {
  responseReturn: 'data',
});

export const baseRequestClient = new RequestClient({ baseURL: '' });
```

修改 `yudao-ui-admin-vben/apps/web-antd/.env.production`：

```env
VITE_GLOB_API_URL=/admin-api
```

### 11.3 nginx 路径代理（关键）

yudao 前端的 API 调用分两类路径，nginx 需要分别处理：

| 前端请求路径 | 后端实际路径 | nginx 配置 |
|------------|------------|----------|
| `/admin-api/xxx` | `/admin-api/xxx` | `location /admin-api/` + `proxy_pass .../admin-api/` |
| `/system/xxx` | `/admin-api/system/xxx` | `location /system/` + `proxy_pass .../admin-api/system/` |

**nginx.conf 完整配置（vben-admin-a）：**

```nginx
server {
    listen 8080;
    root /usr/share/nginx/html;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /admin-api/ {
        proxy_pass http://yudao-server-a:48080/admin-api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /system/ {
        proxy_pass http://yudao-server-a:48080/admin-api/system/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

> ⚠️ **注意**：`location /system/` 的 `proxy_pass` 必须是 `.../admin-api/system/`（末尾带 `/system/`），而不是 `.../admin-api/`。nginx proxy_pass 规则：
> - `location /system/` + `proxy_pass .../admin-api/` → 路径变成 `//admin-api/system/xxx`（双斜杠，404）
> - `location /system/` + `proxy_pass .../admin-api/system/` → 路径变成 `/admin-api/system/xxx`（正确）

### 11.4 公共接口（免登录）

这些接口不需要 token，返回 401 说明路径代理有问题：

- `GET /system/tenant/get-by-website?website=xxx` — 根据域名获取租户
- `POST /system/captcha/get` — 获取验证码
- `POST /system/auth/login` — 登录

### 11.5 排查流程

1. **405 Not Allowed**：nginx 没有匹配到该路径的 location（路径写错了）
2. **401 账号未登录**：路径代理错，nginx 到后端路径不对，后端收到错误路径后不认识接口
3. **Refresh token is null**：前端请求了需要登录的接口但无 token，通常也是路径错导致的

```bash
# 本地验证公共接口
curl "http://localhost:8081/system/tenant/get-by-website?website=test.com" -H "tenant-id: 1"
```
